Matching Engine

A production-quality limit order-book matching engine in Java 21, implementing strict price-time priority (FIFO within each price level) for a single instrument. Supports limit and market orders on both sides, partial fills, O(1) cancel-by-id, and live top-of-book / depth queries.

view_source ↗

the_problem

An exchange has to decide, for every incoming order, exactly who trades with whom and at what price — deterministically, in the right order, and fast. The rules are unforgiving: the best price must trade first, ties break by arrival time, prices must be exact (a cent of rounding error is a real loss), and cancels have to be instant. I wanted to build that core engine properly, the way it would be defended in a low-latency trading context.

design_decisions

  • Integer ticks, never doubles: prices are stored as long tick counts, so crossing comparisons are exact and deterministic — no floating-point rounding.
  • TreeMap per side + FIFO levels: bids and asks are red-black trees keyed by price (bids descending, asks ascending), giving O(log P) access to the best price and ordered traversal for market sweeps. Each price maps to a FIFO queue, so time priority is just insertion order.
  • Intrusive linked list for O(1) cancel: each order is a node in a doubly-linked list, and a hash map indexes orders by id — so any order can be located and unlinked in constant time.
  • Trades print at the resting price: the passive order set the price first, so a crossing aggressor receives any price improvement — the standard continuous-matching rule.
  • Deterministic logical clock: the engine assigns ids and timestamps itself, making every run reproducible for testing and replay.

key_features

  • Order types: LIMIT and MARKET, on both BUY and SELL sides.
  • Partial fills: leftover limit quantity rests on the book; leftover market quantity is cancelled.
  • Market data: best bid / ask, spread, and aggregated depth per side.
  • Tested & automated: 48 JUnit 5 tests (including randomized quantity-conservation and never-crossed-book invariants) run in GitHub Actions CI on every push.

what_i_learned

This project made the cost of every data-structure choice concrete: why a TreeMap plus intrusive linked list beats a naive list once you care about the complexity of each operation, why integer arithmetic is non-negotiable for correctness, and how much a deterministic clock simplifies testing. I also learned to treat performance as something you measure honestly — warming up the JIT, reporting percentiles, and being candid about timer overhead — rather than something you claim.
